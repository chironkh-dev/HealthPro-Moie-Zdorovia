# План логіки роботи крокоміра для Android (Samsung Doze-стійкий)

## 1) Цілі сервісу
- Підрахунок кроків має працювати 24/7 навіть при вимкненому екрані.
- Дані не мають втрачатися після Doze, перезавантаження, kill процесу або апдейту застосунку.
- На Samsung (One UI) рішення має бути «не вбиваємим» в межах можливостей Android-платформи.

## 2) Базова архітектура

### 2.1 Джерела даних
1. `TYPE_STEP_COUNTER` (основне джерело):
   - дає кумулятивний лічильник кроків від останнього boot.
   - енергоефективний, працює через Sensor Hub.
2. `TYPE_STEP_DETECTOR` (fallback):
   - подія на кожен крок, якщо `STEP_COUNTER` недоступний або глючить.

### 2.2 Життєвий цикл компонентів
- `ForegroundService` (мінімальний пріоритет повідомлення, але постійний):
  - підписка на сенсор;
  - heartbeat self-check кожні 15-30 хв.
- `BootReceiver`:
  - перезапуск сервісу після reboot.
- `PackageReplacedReceiver`:
  - підйом сервісу після оновлення app.
- `WorkManager` (Periodic + OneTime expedited):
  - контрольна перевірка, що сервіс живий;
  - реєстрація сенсора після edge-case падінь.

## 3) Модель даних (стійкість до втрати процесу)

Зберігати в локальній БД (Room):
- `boot_id` (або `boot_timestamp`),
- `sensor_baseline` (значення `STEP_COUNTER` на старті доби),
- `last_sensor_value`,
- `daily_steps`,
- `last_event_time`,
- `service_health_state`.

Правило розрахунку:
- `delta = current_sensor_value - last_sensor_value`
- якщо `delta < 0` → вважаємо reboot/overflow, ініціалізуємо новий baseline.
- `daily_steps += max(delta, 0)`

Запис робити батчами (напр. кожні 20-40 кроків або 30-60 сек), щоб зменшити I/O.

## 4) Алгоритм запуску та відновлення

1. App стартує → перевірка permission + hardware sensor availability.
2. Підняти `ForegroundService`.
3. У сервісі:
   - прочитати останній стан з БД,
   - перевірити зміни `boot_id`,
   - підписатись на `STEP_COUNTER`.
4. Якщо сенсор не дає подій N хвилин при наявності руху:
   - зробити soft re-register listener;
   - якщо не допомогло — fallback на `STEP_DETECTOR`;
   - логнути інцидент у telemetry.
5. `WorkManager` кожні 15 хв (або системний мінімум):
   - перевірка, що foreground notification існує;
   - перевірка, що `last_event_time` не протермінований;
   - при збою — запустити recovery сценарій.

## 5) Samsung Doze / One UI hardening

1. Запросити користувача додати app в:
   - **Unrestricted battery**,
   - **Never sleeping apps**.
2. Показати guided-екран з інструкцією для One UI (покроково, зі скрінами).
3. На Android 13+ коректно запросити `POST_NOTIFICATIONS`, щоб FGS-нотифікація не зникала.
4. Не залежати від exact alarms для core-логіки крокоміра.
5. Реалізувати «самовідновлення»:
   - boot completed,
   - app update completed,
   - periodic health-check,
   - ручна кнопка «Відновити крокомір» у налаштуваннях.

## 6) UX та прозорість

- Статус крокоміра: `Active / Limited / Paused by system`.
- Діагностика в app:
  - sensor available,
  - foreground running,
  - battery optimization status,
  - last sensor event time.
- Якщо система обмежує app — показати чіткі кроки виправлення.

## 7) Аналітика та моніторинг якості

Логувати події:
- `pedometer_started`, `pedometer_recovered`, `sensor_timeout`, `battery_opt_blocked`.
- Метрики:
  - % часу активного збору,
  - середній downtime,
  - частота recovery на Samsung vs інші бренди.

## 8) Тест-план (обов'язково)

1. **Doze test**: ADB force-idle + нічний сценарій 8 год.
2. **Process kill test**: свайп із recent + auto-recovery.
3. **Reboot test**: перезапуск девайсу, перевірка continuity.
4. **Update test**: оновлення APK поверх старої версії.
5. **No-motion / motion test**: перевірка false-idle.
6. **Samsung matrix**: One UI 5/6, mid-range + flagship.

## 9) Мінімальні технічні вимоги

- Android 10+
- Foreground Service type: `health` (де доступно) або коректний сумісний type.
- Room + WorkManager + BroadcastReceiver.
- Crash-safe state machine для сервісу.

## 10) Definition of Done

- На Samsung: не менше 99% днів без втрати денного підрахунку.
- Після reboot/update/process-kill сервіс відновлюється автоматично.
- Користувач бачить статус і може самостійно відновити роботу через UI.
