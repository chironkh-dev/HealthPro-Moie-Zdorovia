Аналіз поточного стану бекапу

Що зараз зберігає exportData() (версія '4.0'):
Дані	Статус
measurements (виміри тиску)	✅ є
pills (ліки)	✅ є
pillsTaken (прийняті ліки)	✅ є
settings (часткові)	⚠️ неповні
steps_log (кроки по днях)	❌ відсутній
theme (тема)	❌ відсутня
lang (мова)	❌ відсутня
bpStandard (ESC/AHA)	❌ відсутній
biometricLock (блокування)	❌ відсутній
med_taken (SQLite-таблиця)	❌ не покривається

Головна проблема: бекап читає state (localStorage-дзеркало), а не SQLite — тому дані кроків, нові налаштування та med_taken з реляційних таблиць взагалі не потрапляють у файл.
Пропозиція нового алгоритму бекапу v2.0
Принципи

    Читати з SQLite (не зі state) — єдине джерело правди
    Checksum — SHA-256 для перевірки цілісності файлу
    Schema version — для автоматичної міграції при відновленні
    biometricLock — НЕ відновлювати автоматично — питаємо користувача після відновлення (безпека)
    SQLCipher ключ — НІКОЛИ не включати у файл (він у Android Keystore)
    Формат файлу — .json з розширенням healthpro-backup-ДАТА.json

Структура файлу бекапу (Schema 2)

{
  "hp_backup_version": "2.0",
  "app_version": "5.2.000",
  "schema": 2,
  "created_at": "2026-05-08T12:00:00.000Z",
  "checksum": "sha256-abc123...",
  "data": {
    "measurements": [
      { "sys": 120, "dia": 80, "pulse": 72, "note": "", "ts": 1746700000 }
    ],
    "medications": [
      { "id": "abc", "name": "Лозартан", "dose": "50 мг",
        "time": "08:00", "days": "daily", "date": "" }
    ],
    "med_taken": [

}

    biometricLock у бекап не входить — після відновлення додаток запитає чи увімкнути блокування знову

Алгоритм ЕКСПОРТУ (збереження)

1. Зчитати всі таблиці з SQLite (measurements, medications, med_taken, steps_log, kv_state)
2. Зібрати settings — об'єднати defaultSettings + kv_state (без biometricLock/pinCode)
3. Додати theme, lang
4. Серіалізувати у JSON
5. Порахувати SHA-256 checksum від data-блоку → записати у поле checksum
6. Завантажити файл: healthpro-backup-YYYY-MM-DD.json

Алгоритм ІМПОРТУ (відновлення)

1. Зчитати файл → перевірити checksum (якщо не збігається → помилка, не продовжувати)
2. Перевірити schema version → якщо schema=1 (старий формат) → трансформувати
3. Показати модалку підтвердження:
   "Буде відновлено: N вимірів, M ліків, K днів кроків. Поточні дані буде замінено."
4. Якщо підтверджено:
   a. Очистити всі 5 таблиць SQLite
   b. Записати measurements, medications, med_taken, steps_log
   c. Записати settings у kv_state (без biometricLock)
   d. Записати theme, lang
5. Показати: "Відновлено! Блокування екрану вимкнено — увімкніть у Налаштуваннях"
6. location.reload()

Питання безпеки бекапу (опція для узгодження)

Зараз файл — відкритий JSON (дані видно у будь-якому редакторі). Є два варіанти:
Варіант	Плюси	Мінуси
А. Відкритий JSON (поточний підхід)	Простий, сумісний, читабельний	Медичні дані видно без захисту
Б. Шифрований бекап (AES-256-GCM, пароль вводить користувач)	Медичні дані захищені	Треба пам'ятати пароль; складніше у реалізації
